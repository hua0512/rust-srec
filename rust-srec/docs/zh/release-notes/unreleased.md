# 更新日志

## `unreleased`

本次更新重写了负责录制 HLS 流的 **Mesio** 引擎，让录制更稳定、更可靠，也让 HLS 与 FLV 录制的处理方式趋于一致。加密流录制更可靠，繁忙或加密流的内存占用保持可控；当直播刷新访问链接（例如 Twitch）时，应用也不会再把同一段内容重复下载。一些已不再影响录制的引擎设置被移除，你现有的设置会自动更新。

FLV 与 HLS 文件的录制与修复也降低了 CPU、内存和磁盘开销。应用只对每条流分析一次，而不是反复分析；在超大流上也能控制内存占用；录制结束时不再重写整个文件即可完成收尾。更新的流格式，以及各种被截断或损坏的输入，也能更从容地处理。

斗鱼也做了一处小而实用的改进：现在可以直接在清晰度选择器里选择仅音频流，H.265（HEVC）流的识别更准确，更多“房间不可用”响应会被当作单纯离线处理，而不是报成错误。

控制台的主题系统也翻新了一遍。现在无论网页端还是桌面端，深色模式和自定义主题都会在页面显示出来之前就准备好——用深色模式的用户打开桌面应用，再也不会先白屏一闪了。明暗切换也比以前更顺滑、更稳定。

## 平台

- **新增平台：SOOP**

  现在可以直接添加 SOOP（前 AfreecaTV，`play.sooplive.co.kr` / `play.sooplive.com`）主播，并按你选择的清晰度录制。密码锁房与需要登录的直播可通过平台设置或 Cookie 支持；配置了 SOOP 账号后，应用会自动保持登录有效，供后续检测复用。开启弹幕后可记录公屏聊天与礼物。

- **新增平台：Bigo Live**

  现在可以直接添加 Bigo Live（`bigo.tv`）主播并录制。密码锁房可通过平台设置或主播 URL 中的 `?pwd=` 参数支持；开启弹幕后可记录公屏聊天与礼物。默认情况下，应用会以与其网页相同的方式与 Bigo 通信，以获得更好的兼容性。公开接口下为单一竖屏中等清晰度画面（大约 480p–540p），仅在浏览器中可用的更高清晰度这里不提供。

## 桌面应用

- **深色模式下启动不再白屏一闪**

  以前主窗口会在主题还没来得及应用时就显示出来，用深色模式的话，每次打开应用都会先白一下。现在主题会在窗口出现之前就位；启动画面也会跟着你在应用里选的主题走，而不是只看系统设置——比如系统是浅色、但你把应用设成了深色，那么从启动画面到进入应用，看到的一直都是深色。

- **Windows GPU 检查不再闪现控制台窗口**

  在配有 NVIDIA GPU 的 Windows 系统上，桌面应用此前可能会在启动时以及每次后台 GPU 健康检查运行时短暂闪现黑色控制台窗口。GPU 检查现在不会再打开控制台窗口，启动时也不再执行一次多余的连续检查。

## 主题与外观

- **自定义主题一打开页面就生效**

  保存过的主题预设、导入的主题、自己调过的颜色，现在都会在页面显示之前恢复好，网页端和桌面端都一样，打开时不会再闪一下默认配色。升级新版本之后，只要这次更新没有改动主题本身，你的主题也照常立刻生效。

- **明暗切换更顺滑、更可靠**

  切换明暗时的圆形展开动画，之前偶尔会整个屏幕先变了色、动画才慢半拍地播出来，现在不会了。顶栏的切换按钮也改成按你眼下看到的样子来切换——模式设为**跟随系统**时，第一下点击就有反应，不用再点两次。另外，如果切换前后外观其实没有变化（比如系统本来就是深色，你从**深色**切到**跟随系统**），就只记住你的选择，不再白白播一遍动画。

- **主题设置在多个标签页之间保持同步**

  在一个标签页里换了预设、调了颜色或圆角，其他开着的标签页会立刻跟着变，和明暗模式一直以来的表现一样。

- **删除确认按钮的文字颜色修好了**

  之前因为缺了一条样式，删除模板、通知渠道或工作流时，确认按钮上的文字颜色显示不正确，现在已经恢复正常。

## 录制时间线

- **HLS 不连续点的标识更准确**

  媒体时间线现在能区分“流本身发出的不连续信号”（例如直播中途编码发生变化）与“录制真正缺失了一段内容”。前者会标记为“不连续”，而不是“中断”；某段录制被分割的原因会显示在它实际发生的位置；较长的会话也会加载完整历史，而不是只显示最近一页。

## 录制回放

- **拖动进度不再让画面卡死**

  观看录制时拖动进度（包括全屏或最大化视图），画面不再停在黑屏上不动。如果某次拖动确实卡住，播放器会自动恢复，只有在仍然无法继续播放时才会提示错误。观看直播不受影响。

- **录制弹幕的时间、昵称、礼物与醒目留言都能正确显示**

  回放录制弹幕时，时间和发送者昵称现在都能正确显示，礼物与醒目留言（Super Chat）也会和聊天消息一起显示。较大的弹幕文件现在会在后台加载，因此打开长录制的弹幕不再卡住页面。从 B 站 XML 文件导入的弹幕仍然可用。

## HLS 录制引擎

- **重写 HLS 引擎，录制行为更可预测**

  负责录制 HLS 流的引擎经过重写，录制更稳定、更可预测。它在高负载下的表现更一致，并能控制内存占用（包括加密流所需的额外处理），因此高速流或加密流不会再让内存无限增长。

- **加密 HLS 录制更可靠**

  即便加密数据突发涌入，加密流也能保持流畅，而不会卡顿堆积。对于分片传输的流，应用会确保每个片段所需的初始化数据先写入，避免产生损坏的文件；如果这份初始化数据始终无法获取，录制会显示一个明确的缺口，而不是卡死。

- **会轮换访问链接的流不再重复下载**

  当直播刷新链接、在 URL 中更换签名或访问令牌时，已经下载过的部分不会再被重复抓取。对于链接规则已知的来源（例如 Twitch 签名链接），应用能在多次刷新间识别出同一段内容；下载途中过期的链接也会自动改用新链接重试。

- **缺失的数据会显示为明确的缺口**

  当直播继续推进、某些内容在被下载前就已错过时，录制会标出一个明确的缺口，而不是悄无声息地卡住，从而让缺失的数据能被看到，而不是表现为录制画面冻结。

## FLV 与 HLS 媒体处理

- **更低的处理开销与有界内存占用**

  对 FLV 与 HLS 录制的分析和修复现在只做一次并复用，而不再重复进行；应用只读取每条流所需的部分，并分批处理。处理期间内存始终有上限，因此少数超大片段不会让内存膨胀。如果分片录制在其初始化数据到达前就达到了这个上限，应用会先另起一个输出文件再继续，以保证录制文件有效。

- **FLV 录制收尾更快**

  FLV 录制现在会在开始时预留出摘要信息的空间——仅音频录制以及没有跳转索引的录制也包括在内——并在录制结束时填入，而不再重写整个文件。这减少了磁盘操作，也消除了大文件录制结束时的明显停顿。（旧的低延迟命令行选项仍然可用，但录制现在始终采用更快的就地方式。）

- **FLV 修复对异常输入更稳健**

  更新的视频格式现在能正确识别，文件末尾的最后内容不再被丢弃，空的数据片段能安全处理，内部缓冲也有了上限。这些改进让新格式流的分割与摘要生成更好，同时避免损坏或异常的录制在修复时占用无限内存。

## Mesio 下载器

- **统一 HLS 与 FLV 下载会话**

  HLS 与 FLV 录制现在在底层以相同方式工作，因此无论某条流用的是哪一种，进度、重试和停止录制的行为都保持一致。

- **精简 Mesio 引擎设置**

  引擎设置表单中移除了若干已不再影响录制的 Mesio 设置，包括整个 **性能（Performance）** 标签页，以及几个已不再起作用的高级字段。你保存的设置会自动更新——无需任何手动操作，其余的超时、重试、解密密钥与缺口跳过设置照常生效。

## 斗鱼直播流

- **仅音频选项合并到清晰度选择器中**

  斗鱼的清晰度设置现在把 **仅音频** 放在了同一个选择器里。需要时仍然可以手动输入自定义斗鱼 rate，常用的原画、高清、低清以及 AAC 音频则可以直接从列表中选择。

- **H.265 识别更准确**

  斗鱼直播流现在会根据斗鱼自己服务器返回的信息来标记 H.265（HEVC），并自动请求对应的流，若该流不可用则回退到普通流。不需要单独维护 H.265 开关，仅音频请求也会继续使用 AAC 音频流。

- **更多房间不可用响应会被正常处理**

  更多斗鱼“房间不可用”的响应现在会被当作房间只是离线来处理。这样在房间关闭、主播在检测过程中下播，或斗鱼临时返回不可用响应时，就不会再报成错误。

## 主播管理

- **筛选与排序主播列表**

  主播页面重新设计了筛选工具栏，便于处理数量较多的主播列表。可以按所属模板筛选（包括未分配模板的主播）、按优先级筛选，以及按状态筛选（例如只显示当前处于错误状态的主播），并可按名称、优先级、状态或最近更新时间排序。筛选与排序会作用于整个列表，而不只是你当前查看的这一页，因此计数与页码始终正确。工具栏在桌面端保持紧凑，在较小屏幕上则切换为移动端布局。

- **批量操作多个主播**

  现在可以一次选中多个主播并同时操作。进入选择模式后，可用鼠标或键盘勾选主播，也可以一键选中当前整页，然后对所选主播批量执行启用、禁用、分配模板、设置优先级（高、普通、低）或删除。每个主播都会独立处理：即使部分失败，其余仍会完成，每个失败都会逐个上报，失败的主播还会保持选中，方便你无需重新勾选即可重试。

## 流水线 DAG 编辑器

- **无需重新连线即可替换步骤**

  流水线的列表视图和图表视图现在都提供**替换步骤**操作。选择其他任务预设或子工作流后，只会替换该步骤所做的事情，而它的位置以及与其他步骤的所有连接都保持不变。

- **删除步骤不再遗留无效依赖**

  删除步骤时，它后面的步骤现在会自动连接到它前面的步骤，不再保留指向已删除步骤的连接。删除后再添加步骤时也会获得各自唯一的标识，避免出现重复步骤或含糊的连接。

- **步骤再多也不会被截断**

  流水线步骤较多时，现在可以直接在编辑区域内向下滚动，查看和调整所有步骤，不会再出现底部步骤被遮住的情况。图表视图也会保持在编辑区域内，不再超出边界。

- **分支较多的流水线图表更清晰**

  流水线图表现在能更清晰地呈现分叉与汇合的步骤：步骤之间的连线重叠更少、交叉更少，箭头也更醒目，更容易看清哪个步骤连向哪个步骤。这只改变图表的显示方式，不会改变流水线的实际执行。

- **手动摆放的步骤会保持位置**

  在图表中手动拖动摆放的步骤，现在会在你添加或删除其他步骤时保持原位，而不再每次改动都弹回。新添加的步骤会出现在与它相连的步骤旁边，且不会压在其他节点上；需要恢复自动排布时，随时可以使用**自动布局（Auto Layout）**一键整理。

## 流水线上传

- **上传目标路径支持按会话开始时间锚定日期**

  rclone 流水线步骤新增 `time_anchor` 设置。可以保持默认的 `job_created` 行为，也可以选择 `session_start`，让 `%Y/%m/%d` 等日期占位符使用直播会话的开始时间——这样跨午夜的直播仍会保存在同一个日期远端目录中。copy/move 步骤也可以选择 `job_created` 或 `session_start`，默认仍按步骤运行的时刻展开。

- **上传目标日期按服务器本地时区渲染**

  上传目标路径中的日期占位符现在会按服务器本地时区展开。此前使用的是 UTC，当服务器未设为 UTC 时，可能导致上传所在的日期目录与录制文件名自身的日期不一致。每个日期都会采用正确的时区偏移，因此夏令时切换前后的录制仍保持与文件名一致的日期，重试的上传也会落在与原次相同的日期目录中。

## 流水线步骤表单

- **帮助文本中的占位符名称恢复显示**

  部分流水线步骤表单的帮助文本中占位符名称显示为空白——例如弹幕转换步骤显示“可使用 和 占位符”，而看不到 `{input}` 与 `{output}`。rclone 与 copy/move 的目标路径字段现在也会在表单中直接列出支持的占位符。

- **copy/move 步骤配置无效时报错更清晰**

  当 copy/move 步骤保存的配置无法解析时（例如选项值拼写错误），现在会报告实际的解析错误，而不是误导性的“未指定目标目录”提示。

## 录制可靠性

- **下载启动连续失败后，录制可自动恢复**

  当直播的服务器短暂拒绝下载（例如在直播间仍在线时返回“未找到”错误）时，连续三次失败会让主播进入短暂的冷却。此前的一个时序问题可能导致直播间显示为“在线并报错”、但实际上并没有在录制，直到你手动关闭再打开监控。现在遇到这种情况录制会被干净地停止，并在冷却结束时安排一次新的检查，因此流恢复可达后录制会自动继续。

- **直播看门狗会重启悄然消失的下载**

  作为一道兜底，周期性检查现在会注意到“直播间已显示在线五分钟、却没有任何录制”的情况，并重新开始录制，而不是忽略它。

- **禁用主播不会再被进行中的状态检查“撤销”**

  此前在某次检查仍在进行时禁用主播，该检查可能在你禁用之后又把主播重新标记为在线并重新开始录制。现在，一旦禁用被保存就始终生效——迟到的检查结果会被丢弃，不会覆盖你的选择。

- **录制与其弹幕文件和时间线保持对应**

  每个录制现在都能稳定地与它的弹幕文件以及在录制时间线上的条目保持对应。在 Windows 上，以及把录制保存到链接或快捷方式文件夹时也同样可靠——这些情况下以前可能出现弹幕文件或时间线条目对不上的问题。

## 数据库维护

- **统一且更彻底的自动清理**

  应用现在会自动清理自己的数据库，避免它随时间不断变大。它会清除旧的流水线历史、已完成和已取消的任务、过期的登录、旧的通知记录，以及从未录到任何内容的空录制。应用启动时和每半小时会做一次快速清理，较重的清理则会等到你设定的维护时段再进行。清理是分小批进行的，因此即使积压较多，也会逐步处理，而不是一次性集中完成，磁盘空间也会随时间释放出来。近期或仍在使用的内容始终会保留。

- **保留天数设为 0 表示“永久保留”**

  **流水线历史保留（Pipeline History Retention）** 与 **通知日志保留（Notification Log Retention）** 设置现在会把 `0` 视为“永久保留”。任意正数表示保留该历史的天数；设为 `0` 则会一直保留，直到你自行删除。
